# liberado-qdrant-mcp — Design & Implementation Plan

A lightweight, resource-efficient RAG system exposed as an MCP server that talks
directly to Qdrant through a shared Rust core library. Built for personal /
homelab single-user use, with a clean migration path to a web frontend.

> **Naming:** the project is **`liberado-qdrant-mcp`**. Use `lqm` only as a
> shorthand (binary names, crate prefixes, docs shorthand). Do **not** call it
> "myrag" anywhere.

> **Status (2026-07-18):** P0–P4 and hybrid/scope (part of P5) are **shipped**.
> Treat milestone sections below as historical design intent; live status is
> [`ROADMAP.md`](ROADMAP.md) and the
> [AnythingLLM gap map](../liberado-qdrant-mcp_vs_AnythingLLM_Analysis_and_Implementation_Roadmap.md).

---

## 1. Goals & Non-Goals

### Goals
- **Zero daemon by default.** No persistent process consuming RAM/CPU when idle.
- **Resource efficient.** Small static binary, fast startup, low memory.
- **Runs anywhere.** Target diverse hardware: Raspberry Pi, mini-PC, gaming PC.
  Model choice and concurrency are user-customizable.
- **Simple.** A core library + thin binaries. Qdrant is the only persistent
  service (already run via Docker in the homelab).
- **Direct MCP → Qdrant** via a shared core lib (no extra backend hop).
- **Loose coupling & decomposition.** Multiple crates with clear seams so new
  ingestion sources / embedders / transports drop in without rewrites.
- **Future-proof.** Add a web frontend (Dioxus) later without duplicating logic.
- **Concurrency-aware.** Handle multiple agents / parallel tool calls gracefully
  without becoming a heavy daemon.
- **Well-tested.** Unit + integration tests added alongside every feature.

### Non-Goals (for now)
- Multi-user auth / multi-tenancy.
- Background re-indexing workers / heavy queuing.
- Persistent in-memory state beyond an optional shared-model HTTP mode.
- Web frontend as a priority — it is a nice-to-have; the MCP server comes first.

These are only revisited if a concrete need appears.

## 2. Scope: What We Ingest & Search

**Fundamentally a text search system.** v1 focuses on text-native sources:

- Obsidian vaults / markdown notes
- Code repositories
- Conversation history
- Plain text / docs

**Loose coupling for future mediums.** The ingestion layer is designed so other
media are easy to add later by converting them to text first:

- PDFs → text extraction
- Audio (podcasts, voice notes) → transcription → text
- Anything else → a source-specific extractor that emits text + metadata

The core never assumes a medium; it consumes `DocumentChunk`s produced by
pluggable extractors.

## 3. Architecture Decision

**Chosen: Core library + thin binaries, MCP talks directly to Qdrant.**

MCP servers in the Rust ecosystem are designed around **stdio transport**: the
agent host (Claude Code, Kilo, RustyClaw/OpenClaw, Cursor, etc.) spawns the
binary as a short-lived child process, performs the JSON-RPC handshake and tool
calls over stdin/stdout, then the process exits.

This gives:
- Zero daemon by default.
- Extremely low resource use.
- Clean security model (runs with user permissions, sandboxable by the agent
  framework).

**Rejected (for now): MCP → separate backend binary (HTTP/gRPC).**
Adds an extra hop, extra process, extra latency, more moving parts. Only worth it
for persistent in-memory state, bulk-ingestion queuing, multi-user auth, or
background workers — none required for single-user use.

**MCP SDK: [`turbomcp`](https://github.com/Epistates/turbomcp) (mandated).**
A full-featured Rust MCP SDK (v3.1.x). Rationale in `docs/DECISIONS.md`.
- Zero-boilerplate macros: `#[server]`, `#[tool]`, `#[resource]`, `#[prompt]`,
  `#[description]`.
- Feature-gated transports: `stdio` (default), `http`, `tcp`, `unix`,
  `websocket`, `channel` — matches our dual-mode plan and gives us the runtime
  transport switch essentially for free.
- `run_stdio()` / `run_http(addr)` / `run_tcp()` / `run_unix()` generated by the
  `#[server]` macro.
- In-process `channel` transport + `McpTestClient` → fast, hermetic integration
  tests without spawning processes.
- Requires **Rust 1.89+ / Edition 2024**, Tokio runtime.

### Crate responsibilities

**`lqm-core`**
- All the real work: chunking strategy, embedding orchestration, Qdrant
  interactions, payload schema, filtering logic, concurrency control.
- Pure library — no MCP, no HTTP, no CLI, no side effects except what is
  explicitly called.
- Fully configurable: chunk size/overlap, embedding model/dimension, collection
  defaults, metadata enrichment.
- Async (`tokio`) and batching-friendly throughout.
- Easy to test in isolation.

**`lqm-ingest`**
- Source extractors that turn files/streams into `DocumentChunk`s + metadata.
- Text-native extractors first (markdown, code, plain text, conversation logs).
- PDF / audio-transcription extractors behind optional cargo features later.
- Depends on `lqm-core` types; no MCP/HTTP knowledge.

**`lqm-mcp`** (binary)
- Depends on `lqm-core` (+ `lqm-ingest`).
- Uses `turbomcp` with `#[server]` / `#[tool]` macros.
- Stdio transport by default; `http`/others feature-gated for dual-mode.

**`lqm-cli`** (binary)
- Bulk ingestion, admin commands, collection management via `clap`.
- Also useful for benchmarking embedders.

**`lqm-api`** (shipped binary)
- HTTP server (axum) reusing `lqm-core`. Powers the Dioxus web UI.
- Qdrant remains the source of truth; core enforces consistent chunking/metadata
  so web UI and agents see the same high-quality data.

## 5. Core Design Sketches

### Types & traits (indicative)

```rust
// types.rs
pub struct CollectionConfig {
    pub name: String,
    pub vector_dim: usize,
    pub distance: Distance,        // Cosine by default
}

pub struct DocumentChunk {
    pub text: String,
    pub payload: Payload,
}

pub struct SearchResult {
    pub text: String,
    pub score: f32,
    pub payload: Payload,
}

// embedding.rs — one trait, many adapters (feature-gated)
#[async_trait]
pub trait Embedder: Send + Sync {
    async fn embed_batch(&self, texts: Vec<String>) -> Result<Vec<Vec<f32>>>;
    fn dimension(&self) -> usize;
    fn id(&self) -> &str; // for logging / benchmarking
}
```

### Concurrency + batching (do this first)

```rust
use std::sync::Arc;
use tokio::sync::Semaphore;

pub struct RagCore {
    embedder: Arc<dyn Embedder>,      // fastembed (default) or other adapters
    qdrant: Qdrant,
    embed_semaphore: Arc<Semaphore>,  // limit concurrent embedding
}

impl RagCore {
    pub async fn embed_and_upsert_batch(
        &self,
        texts: Vec<String>,
        metadata: Vec<Payload>,
    ) -> Result<()> {
        let _permit = self.embed_semaphore.acquire().await?; // limits concurrency
        let embeddings = self.embedder.embed_batch(texts).await?; // one efficient call
        self.qdrant.upsert_batch(embeddings, metadata).await?;
        Ok(())
    }
}
```

- Semaphore size is **configurable** (default `num_cpus::get()`, lower it to 1–2
  on a Raspberry Pi) to avoid CPU thrash from bursty concurrent agents.
- **Searches bypass the semaphore** — they are I/O bound to Qdrant and cheap.
- Embedding is batch-oriented: give the model 8–32 texts at once, not one-by-one.

## 6. Payload Schema (Qdrant)

Designed for good filtering later. Add payload indexes on commonly filtered
fields (`collection`, `tags`, `source_type`, `project`).

```json
{
  "text": "chunk content...",
  "source": "/path/to/file.md",
  "source_type": "obsidian",   // | "code" | "conversation" | "pdf" | "audio" | ...
  "collection": "homelab",     // | "work-notes" | ...
  "tags": ["networking", "rtl-sdr"],
  "timestamp": "...",
  "project": "..."
}
```

Payload fields (beyond `text`) should be extensible so future source types can
attach their own metadata without breaking existing filters.

## 7. MCP Tools

- `ingest_path` — ingest a file/directory path.
- `ingest_text` — ingest raw text with metadata.
- `ingest_many(texts: Vec<String>, metadatas: Vec<Json>)` — batch-friendly.
- `search` — vector search with payload filters (collection, tags, min_score…).
- `list_collections`
- `get_collection_info` (implemented; older drafts said `collection_info`)
- Admin tools (create/delete collection, reindex) as needed.

**Design tools for batching** so agents prefer batch calls over one-by-one.

## 8. Dual-Mode Server (concurrency sweet spot)

turbomcp exposes transports as cargo features; the same binary can run stdio or
HTTP based on a flag/subcommand (turbomcp supports runtime transport selection).

| Scenario                          | Mode            | Concurrency handling                   | Resource profile          | Recommendation |
|-----------------------------------|-----------------|----------------------------------------|---------------------------|----------------|
| Casual / single agent             | Stdio           | Multiple short-lived processes         | Very low (spiky)          | Default        |
| Multiple agents / parallel calls  | HTTP `serve`    | Async tasks in one process + semaphore | Low & stable (one model)  | Use when needed|
| Very heavy bulk ingestion         | HTTP + batching | Batching + limited semaphore           | Controlled                | Preferred      |
| Searches (any concurrency)        | Either          | Qdrant + async client                  | Negligible                | No issue       |

- **Stdio** (`lqm-mcp` default): ultra-low resource, spawn-on-demand. Model
  loaded fresh per invocation — fine for bursty/sporadic use.
- **HTTP / Streamable HTTP** (`lqm-mcp serve --port 8080`): one lightweight
  persistent process. Embedding model loaded **once** into RAM; all concurrent
  agents handled as async tasks. Typically tens of MB RSS.
- turbomcp features: `stdio` (default) always on; enable `http` for the
  persistent mode. `channel` is used for in-process integration tests.
- Can be started on-demand (script or user systemd unit).

## 9. Embedding Strategy

Embedders are pluggable behind the `Embedder` trait and wired as **optional
cargo features** so users pick what they install/benchmark.

- **v1 default: [`fastembed`](https://crates.io/crates/fastembed)** — in-process,
  no extra server, models cached after first download.
- **Ollama adapter — nice-to-have / roadmap.** Behind a feature, reuses an
  existing Ollama install.
- **Other Rust embedding libraries** — evaluate and add as optional features
  (e.g. `candle`-based embedders). All adapters share the trait and are
  benchmarkable via `lqm-cli`.
- **Customizable model, per hardware.** Model + dimension are config-driven so a
  Pi can use a tiny model while a gaming PC uses a larger one. Keep `vector_dim`
  in sync with the chosen model; a collection is tied to one embedding space.
- Suggested small/fast starting models: `BAAI/bge-small-en-v1.5`,
  `sentence-transformers/all-MiniLM-L6-v2`, `nomic-embed-text`.

**Benchmarking is a first-class use case:** `lqm-cli` should be able to run the
same corpus through multiple embedders and report speed/memory/quality signals.

## 10. Ingestion Strategy

- Start with **markdown / text / code / conversation history** (text-native).
- Add **PDF** (text extraction) and **audio** (transcription → text) later,
  behind optional features, via the `lqm-ingest` extractor interface.
- Make it **idempotent-ish**: hash chunks or track `source` + `last_modified` in
  payload to avoid duplicate/stale entries.
- Chunking tuned per source type (Obsidian notes, code, conversation, docs).

---

## 11. Qdrant Notes

- High-concurrency reads with low overhead — parallel searches are essentially
  free on modest hardware.
- Efficient concurrent batch upserts during ingestion.
- Fast payload filtering once indexes exist on filtered fields.
- Use the async `qdrant-client` (automatic connection pooling in recent
  versions).
- For very heavy bulk loads: optionally disable HNSW indexing during the load and
  re-enable after — or just let it run (Qdrant is forgiving on modest hardware).

---

## 12. Filtering / "Clearance-safe" Behavior

Requirements are not fully defined yet; the guiding principle is
**customizability**. Design so scoped/filtered searches are driven by config +
payload filters (collection, tags, source_type, project) rather than hard-coded
rules. Leave a seam for user-defined filter policies so stricter rules can be
added later without core changes.

---

## 13. Testing Strategy

Tests are added **alongside** features, not deferred.

- **Unit tests** in each crate: chunking boundaries, payload construction,
  filter building, semaphore/batch logic, embedder adapters (with a fake
  embedder for determinism).
- **Integration tests** for `lqm-mcp` using turbomcp's `channel` transport +
  `McpTestClient` (in-process, no spawned binary).
- **Qdrant integration tests** gated behind a feature/env flag, run against a
  local Docker Qdrant.
- A **fake/mock `Embedder`** (fixed vectors) keeps most tests fast and offline.

## 17. Resolved Guidelines (from user)

1. **Scope:** text-first (Obsidian, code, conversation history); loosely coupled
   so PDFs/audio (transcribed) are easy to add. Fundamentally text search.
2. **Crates/binaries:** multiple crates + binaries preferred; loose coupling and
   decomposition are priorities.
3. **Embedding v1:** `fastembed` in-process. Ollama + other Rust embedders are
   optional features (roadmap) and available for benchmarking.
4. **Hardware:** must target diverse hardware (Pi → gaming PC); model choice and
   concurrency are user-customizable.
5. **Filtering:** exact rules TBD; prioritize customizability, no hard-coding.
6. **MCP SDK:** must use `turbomcp` (Epistates/turbomcp).
7. **Frontend:** Minimal static UI shipped; Dioxus SPA remains nice-to-have. MCP + REST remain the priority for agents.
8. **Process:** regular audits for duplication/coupling/decomposition; add unit +
   integration tests as features land.
9. **Docs to maintain:** `docs/ARCHITECTURE.md`, per-crate `ARCHITECTURE.md`,
   `docs/ROADMAP.md`, `docs/DECISIONS.md`, and `AGENTS.md`.
